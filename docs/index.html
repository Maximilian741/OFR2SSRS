The challenge

Migrating off Oracle Reports is painful, manual, and risky

Oracle Reports has been end-of-life for years, but most organizations still have a backlog of business-critical RDF/XML reports tying them to legacy infrastructure.

Hand-translation is slow

Each report — its parameters, datasets, layout, and PL/SQL formulas — has to be rebuilt manually in Report Builder. Weeks per report, multiplied by hundreds.

Dialect drift breaks queries

Oracle SQL — DECODE, NVL, (+) joins, LISTAGG, SYSDATE, lexical refs, package functions — does not survive a copy-paste into T-SQL. Subtle bugs ship to production.

Distribution doesn't transfer

Bursting and DDS were Oracle Reports features. SSRS Standard has no native equivalent, so distribution logic gets lost in the migration unless you rebuild it explicitly.

How it works

Three steps from legacy XML to deployable RDL

A deterministic Python pipeline does the conversion. AI assistance is optional, not required.

1

Drop your artifacts

Upload an Oracle Reports XML — or a folder containing the XML, supporting .docx SQL queries, screenshots, and rendered PDFs. The ingest engine auto-classifies each file.

.xml · .rdf · .docx · .pdf · .png
2

Convert and validate

The pipeline parses, translates PL/SQL to T-SQL, generates SSRS RDL, and runs a pre-flight audit against ten correctness rules. Cross-validation catches parser drift between primary and supporting artifacts.

parse → translate → emit → audit
3

Deploy to SSRS

Inject your connection string at convert-time (never logged, never stored), download the RDL, open it in Report Builder, and publish to your Report Server.

.rdl ready to publish
Capabilities

Everything you need for a confident migration

Built from the ground up to be deterministic where it matters and assistive where it helps.

XML → RDL Generator

Emits well-formed SSRS 2008+ RDL with DataSources, DataSets, ReportParameters, master-detail Tablix, and image controls.

PL/SQL → T-SQL

DECODE, NVL, TO_CHAR, TRUNC, SYSDATE, INSTR, SUBSTR, (+) joins, CONNECT BY, LISTAGG, ROWNUM, lexical refs, bind variables, package functions.

Live data preview

Run translated T-SQL against a seeded SQLite database with Python UDFs that mirror SQL Server scalar functions — see real rows before you deploy.

Pre-flight audit

Ten deterministic rules check parse, namespace, datasources, datasets, Oracle leftovers, parameter consistency, layout integrity, and size units.

Cross-validation

Compares the parsed model against any supplied SQL .docx, screenshots, and rendered PDFs to catch parser drift before download.

Email bursting

Detects bursting patterns in your Oracle layout and emits a burst-query SQL stub plus a PowerShell driver that runs against SSRS Standard via service account.

Architecture

A clean, inspectable pipeline

Each stage is a small, testable Python module. Drop in your own translator rules via the registry plugin system.

Quick start

Run it locally in under a minute

No database required to try it. The seeded SQLite sample lets you see live previews immediately.

Install & run 

# clone the repo
git clone https://github.com/your-org/oracle2ssrs.git
cd oracle2ssrs

# Windows
run.bat

# macOS / Linux / WSL
./run.sh

# Then open
http://127.0.0.1:5057

Convert via API 

# single XML
curl -F file=@report.xml \
     http://localhost:5057/api/convert

# multi-file bundle (XML + docx + screenshots)
curl -F files=@report.xml \
     -F files="@report queries.docx" \
     -F files="@report screenshots.docx" \
     http://localhost:5057/api/convert-bundle

# batch CLI
python backend/cli.py samples/oracle --out ./out

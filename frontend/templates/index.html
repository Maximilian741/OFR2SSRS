Target database backend

Report-server data source (recommended)
Path (or sibling name) of the SHARED data source on your report server. Baked into every generated RDL — main report, sub-reports, burst pack — so uploads bind automatically. No repointing, no refresh dance. Remembered in this browser.
Report server URL (for drill-through links)
Sub-report drill-throughs are always turned into parameterized URL links, so they fire in the SSRS viewer AND in an exported PDF (a plain drill-through is dropped in PDF/Word/Excel). Optional: leave this blank and the link builds itself from the server at run time (Globals!ReportServerUrl + Globals!ReportFolder, zero-config on SSRS 2016+). Fill it in — your report-server URL up to the FOLDER, e.g. http://host/ReportServer?/MVWF — for older servers or to point the links at a specific folder. Use the ReportServer URL, not the /Reports portal URL. The child report must be deployed in that folder under its exact name. Remembered in this browser.
Report images (seals / logos)
Image placeholders detected in the layout. Images embedded in the Oracle export load automatically; upload a PNG/GIF/JPG to fill or replace one — it is embedded into the RDL and shown in the mockup immediately.
Batch migration
Select a whole folder of Oracle XML exports: every report is converted, scored, and bundled with a Migration Assessment (effort per report, upload checks, fidelity).
Embedded connection (optional)
Embeds a connection INSIDE the RDL instead of referencing a shared data source. Sent only with the next conversion, never stored. Prefer the shared path above — embedded connections can make SSRS prompt at design time.

The transformation

Oracle Reports XML in. Uploadable RDL out.

Not a lossy export — a structural translation. SQL dialect, format masks, bind variables, and every layout field are carried across to their SSRS equivalents.

report.xml Oracle
<!-- Oracle Reports XML -->
<report name="WIDGET_ORDERS">
  <dataSource name="Q_ORDERS">
    <select>
      SELECT order_id,
             NVL(amount, 0) amount,
             TO_CHAR(dt, 'MM/DD/YYYY')
        FROM orders
       WHERE region = :P_REGION
    </select>
  <field source="AMOUNT"
         formatMask="$NNN,NN0.00"/>
</report>
→
report.rdl SSRS
<!-- SSRS RDL 2008 — uploadable -->
<Report xmlns="…/reportdefinition">
  <DataSet Name="Q_ORDERS">
    <CommandText>
      SELECT order_id,
             ISNULL(amount, 0) amount,
             FORMAT(dt, 'MM/dd/yyyy')
        FROM orders
       WHERE region = @P_REGION
    </CommandText>
    <Field Name="AMOUNT"/>
  </DataSet>
  <Textbox Name="F_AMOUNT">
    <Value>=Fields!AMOUNT.Value</Value>
    <Format>C2</Format>
</Report>

Real translations in this example: NVL → ISNULL · TO_CHAR → FORMAT · Oracle :bind → @param · formatMask → <Format> · every field becomes an addressable =Fields!… textbox.
Synthetic sample — the engine carries zero per-report data; this is illustrative only.

Why it's different

Built for the parts that other converters skip

Most tools stop at "well-formed RDL." Oracle2SSRS keeps going — distribution, drill-through, and the credentials problem that wastes an afternoon every deploy.

SharedDataSourceReference design load-bearing

RDL is emitted with a DataSourceReference pointed at the shared data source path you configure in the sidebar — no embedded connection string, no credentials block. When the path matches a shared data source on your server, the upload binds automatically. Every parameter carries a =Nothing default and every query bind is wired, so the "Define Query Parameters" dialog never pops — at upload, Refresh Fields, or run.

This is the difference between a 30-second deploy and a 30-minute one.

Verified with Microsoft's rendering engine RenderLab

The repo ships tools/renderlab: a harness that feeds generated RDLs to Microsoft's ReportViewer LocalReport engine — the same processing code SSRS runs — with synthetic data, renders real PDFs, and measures them: page cadence, blank pages, border-rect centering. Publish-time semantic rules that XSD validation can never catch (Lookup/aggregate restrictions, TablixMember consistency) are caught here, before you ever upload.

One command fetches the official Microsoft runtime from NuGet; the rendering checks then run as part of the standard pytest suite.

Bursting that actually ships production

Detects bursting patterns in the source Oracle layout and emits a Burst Pack: SQL stub for the recipient query, a PowerShell driver to render and email per-recipient, and an email config template.

Works against SSRS Standard via a service account — no SSRS Enterprise required.

PL/SQL to T-SQL translation

Handles DECODE, NVL, TO_CHAR, TRUNC, SYSDATE, INSTR, SUBSTR, (+) joins, CONNECT BY, LISTAGG, ROWNUM, lexical refs, bind variables, and package functions.

Output is plain T-SQL you can paste straight into SSMS.

Pre-flight audit + cross-validation

Before download, deterministic rules check parse health, namespace, datasources, datasets, Oracle leftovers, parameter consistency, and layout integrity.

Supporting docs (SQL .docx, PDFs, screenshots) get cross-checked against the parsed model to catch parser drift before it ships.

Trust the output

Upload-safe and faithful — by construction

Two independent self-checks run on every conversion, so you know it will upload to SSRS and that nothing was silently dropped.

🛡️

Will it upload? — the schema gate

Every generated RDL is validated against Microsoft's real RDL 2008 schema — the same definition Report Server uses — plus a pre-flight pass for the object-model rules a schema can't express: duplicate DataSet/Field names, invalid identifiers, parameter-case mismatches.

If it would fail at upload, you see a blocker here first — not a cryptic SSRS error an hour later.

✓

Is it faithful? — the fidelity report

The converter re-parses its own RDL and compares it back to the source: every column, parameter, format mask, and formula is accounted for. You get a per-report fidelity score and an explicit list of anything that still needs manual wiring.

FAQ

Questions, answered

Do I need Oracle installed to use it?

No. It works from the exported Oracle Reports XML (or .rdf / .sql) — no Oracle client, database connection, or license required. Drop the file in and go.

Which SSRS versions can host the output?

The generated RDL targets the RDL 2008 schema, which uploads to SQL Server Reporting Services 2008 R2 through 2016 / 2017 / 2019 / 2022, plus Power BI Report Server. Newer servers host the 2008 definition without complaint.

Is my report data safe?

Yes. The app runs entirely on your machine (a local Flask server) — nothing is sent anywhere. And the project itself ships zero real report data: every fixture and sample is synthetic.

What doesn't convert automatically?

Oracle PL/SQL formula bodies (CF_ / CP_) become wireable placeholders, and package functions get dbo.fn_* T-SQL stubs to finish at deploy time. You're never guessing what's left — the fidelity report lists each item explicitly.

Does it handle bursting and drill-through?

Both. Per-recipient distribution is detected and packaged as a downloadable Burst Pack (recipient SQL, a PowerShell driver, email config). Drill-through child reports are detected and generated as wired-up child RDL stubs.

Is it actually generic, or tuned to specific reports?

Generic. No report names are hardcoded in the conversion engine — a standing test fails if any sneak in. Output is driven entirely by the parsed structure and layout geometry, never a per-report lookup table.
